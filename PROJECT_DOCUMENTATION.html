Project Overview

This system is a Dynamic Multi-Agent AI Pipeline built to automate the processing of PDE (Prescription Drug Event) claims for Medicare Part D. When CMS (Centers for Medicare & Medicaid Services) returns error codes on PDE submissions, this system resolves them automatically using an LLM-driven orchestrator that reads official SOPs and dispatches the right specialist agents.

🎯 What Problem Does It Solve?

Medicare Part D plans submit PDE claims to CMS electronically. CMS returns error codes when claims fail validation. Manual resolution of these errors is time-consuming, error-prone, and requires staff to follow detailed Standard Operating Procedures. This system automates that entire workflow end-to-end.

🔑 Key Features

  • Dynamic orchestration — the LLM reads the SOP and decides which agents to call and in what order for each individual claim
  • No hardcoded routing — all pipeline decisions are driven by GPT-4o reasoning, not static if/else logic
  • Specialist agents as tools — each downstream agent (RX, Report, ServiceNow, Email) is a callable tool within the orchestrator's ReAct loop
  • Two error codes handled — Error 781 (missing/invalid provider) and Error 935 (already adjudicated)
  • Full audit trail — every agent's reasoning trace is captured and displayed in the Streamlit UI
  • Streamlit dashboard — real-time monitoring, claim processing, and database explorer

📌 Supported Error Codes

Error CodeDescriptionBusiness RulePossible Outcomes
781 Provider ID missing or invalid Look up CMS provider mapping table. If found → correct & reprocess. If not found → reject. REPROCESS or REJECT
935 Claim already adjudicated Compare received_date vs adjudication_ts. If equal → duplicate, reprocess. If adj newer → already done, report only. REPROCESS or ALREADY_PROCESSED

Dynamic Orchestration Flow

Unlike a traditional static pipeline where routing is hardcoded, this system uses a Dynamic ReAct Orchestrator. The LLM reads the SOP for each error code, inspects the claim, and decides at runtime which specialist agents to invoke and in what sequence.

📥
INPUT
{ claim_id, error_code }
↓
🧠
DYNAMIC ORCHESTRATOR
LangGraph Node · GPT-4o ReAct Loop
↙
⚙️ Orchestrator's Tool Belt (called sequentially by LLM decision)
① SOP TOOL
tool_load_sop()
reads business rules
② CLAIM TOOL
tool_get_claim()
fetches claim from DB
③ RX AGENT
tool_run_rx_agent()
adjudicates claim
④ REPORT AGENT
tool_run_report_agent()
builds RCL / compliance
⑤ SERVICENOW
tool_run_servicenow_agent()
raises incident ticket
⑥ EMAIL AGENT
tool_run_email_agent()
sends notification
↑ GPT-4o decides which of tools ③–⑥ to call based on SOP + decision ↑
↓
📊
FINAL STATE OUTPUT
decision · agents_invoked · report · rcl_file · ticket · email · reasoning · all traces
🧠 Why is this "dynamic"? The orchestrator does not have hardcoded if/else routing. GPT-4o reads the SOP text and the claim data, then reasons step-by-step about which agents are needed. If it decides a REJECT, it stops after the RX agent without calling report, ServiceNow, or email — purely from reading the SOP rule.

Agent-by-Agent Detailed Explanation

Each agent is a LangChain ReAct agent (create_agent) powered by GPT-4o. They each have a focused system prompt, a set of tools, and return structured JSON output. The orchestrator calls them as tool wrappers.

🧠 1. Dynamic Orchestrator Agent  Supervisor

File: agents/orchestrator.py  ·  LangGraph Node: ORCHESTRATOR

The master controller. Runs the entire pipeline in one ReAct loop. It reads the SOP, fetches the claim, then decides which specialist agents to invoke as tools — collecting all results and writing them into the shared state.

Tools available:

tool_load_sop tool_get_claim tool_run_rx_agent tool_run_report_agent tool_run_servicenow_agent tool_run_email_agent

Shared Context (_ctx): A module-level dict that all tool wrappers read/write so intermediate results (decision, report, rcl_file) flow between agent calls without re-fetching from DB.

Output written to state: decision, agents_invoked, reasoning, report, rcl_file, servicenow_ticket, email_status, plus all 4 sub-agent trace strings.

💊 2. RX Claim Agent  Specialist

File: agents/rx_claim_agent.py  ·  Handles: Error 781 and Error 935

The adjudication specialist. Handles both error codes with different logic paths determined by the system prompt.

  • Error 781 path: fetches claim → looks up provider mapping → updates provider_id in DB → decides REPROCESS or REJECT
  • Error 935 path: fetches claim → compares received_date vs adjudication_ts via date tool → updates status → decides REPROCESS (equal) or ALREADY_PROCESSED (adj newer)

Tools available:

tool_get_claim tool_get_provider_mapping tool_update_claim_provider_id tool_update_claim_status tool_compare_claim_dates

Returns JSON: decision, provider_resolved, new_provider_id, received_date, adjudication_ts, reason

📋 3. Report Builder Agent  Specialist

File: agents/report_builder_agent.py  ·  Invoked when: decision ≠ REJECT

Handles two distinct reporting scenarios based on the decision value:

  • Scenario A (REPROCESS): fetches claim → gets all READY_FOR_REPROCESS claims → generates RCL CSV file → saves audit report to DB
  • Scenario B (ALREADY_PROCESSED): fetches claim → saves compliance report to DB (no RCL file generated)

Tools available:

tool_get_claim tool_insert_report tool_generate_rcl_file tool_get_reprocess_claims_summary

Returns JSON: report, decision, rcl_file (path or null)

🎫 4. ServiceNow Agent  Specialist

File: agents/servicenow_agent.py  ·  Invoked when: decision = REPROCESS only

Creates a simulated ServiceNow incident ticket for the PDE reprocessing team. In production this would call the ServiceNow Table REST API.

  • Gets all READY_FOR_REPROCESS claims from DB
  • Creates ticket with: short description, full description, priority (2-High if >5 claims, else 3-Medium), category = "PDE Reprocessing", affected_claims list
  • Assigns to "PDE Operations Team / Medicare Part D Processing" group

Tools available:

tool_get_reprocess_claims_summary tool_create_servicenow_ticket

Returns JSON: ticket_number (e.g. INC20260303124054), ticket_summary, claims_count

📧 5. Email Agent  Specialist

File: agents/email_agent.py  ·  Invoked when: decision ≠ REJECT

Composes and (simulated) sends a professional notification email to the compliance team. In production this would connect to SMTP / SendGrid / Azure Communication Services.

  • Fetches claim to confirm final status
  • Composes a 3–5 sentence professional email including claim ID, error code, decision, and key details
  • Sends to compliance-team@healthplan.com

Tools available:

tool_get_claim tool_send_email

Returns JSON: email_status (SENT), email_subject, email_summary

All 4 Scenarios — Complete Walkthrough

✅ Scenario 1 — Error 781: Missing Provider → REPROCESS

Claim: C0009 | provider_id = '' (blank)
  1. Orchestrator loads SOP 781: "retrieve correct provider ID from RxClaim"
  2. Fetches claim C0009 — provider_id is blank
  3. Calls RX Agent → looks up mapping for '' → finds P002 (Sunrise Pharmacy)
  4. RX Agent updates DB: provider_id = P002, status = READY_FOR_REPROCESS
  5. Decision = REPROCESS
  6. Calls Report Agent → generates RCL CSV file with all reprocess claims
  7. Calls ServiceNow Agent → raises incident ticket (priority 2-High)
  8. Calls Email Agent → sends notification to compliance-team@healthplan.com

🔄 Scenario 2 — Error 935: Dates Equal → REPROCESS

Claim: C0010 | received=2025-01-02, adj_ts=2025-01-02
  1. Orchestrator loads SOP 935: "compare claim received date with adjudication timestamp"
  2. Fetches claim C0010 — dates are equal
  3. Calls RX Agent → runs date comparison tool → dates EQUAL → potential duplicate
  4. RX Agent updates status = READY_FOR_REPROCESS
  5. Decision = REPROCESS
  6. Calls Report Agent → generates RCL CSV
  7. Calls ServiceNow Agent → raises ticket
  8. Calls Email Agent → sends notification

ℹ️ Scenario 3 — Error 935: Adj Newer → ALREADY_PROCESSED

Claim: C0006 | received=2024-12-15, adj_ts=2025-01-02
  1. Orchestrator loads SOP 935: "if adjudication newer → mark as already processed"
  2. Fetches claim C0006 — adj_ts is 18 days after received_date
  3. Calls RX Agent → date comparison: adj NEWER → no reprocess needed
  4. RX Agent updates status = ALREADY_PROCESSED
  5. Decision = ALREADY_PROCESSED
  6. Calls Report Agent → saves compliance report (no RCL file)
  7. ⛔ ServiceNow Agent is NOT called (SOP says: exclude from resubmission)
  8. Calls Email Agent → sends notification

❌ Scenario 4 — Error 781: Invalid Provider → REJECT

Claim: TEST-REJECT-001 | provider_id = P999 (no mapping)
  1. Orchestrator loads SOP 781
  2. Fetches claim TEST-REJECT-001 — provider_id = P999
  3. Calls RX Agent → looks up mapping for P999 → no mapping found
  4. RX Agent updates status = REJECTED
  5. Decision = REJECT
  6. ⛔ Report Agent is NOT called
  7. ⛔ ServiceNow Agent is NOT called
  8. ⛔ Email Agent is NOT called
  9. Pipeline ends — only RX_AGENT was invoked

Standard Operating Procedures (SOPs)

The SOPs are plain text files in the sop/ directory. The orchestrator loads them at runtime using tool_load_sop(error_code) and passes the text to GPT-4o, which uses it to reason about what actions to take. This means adding a new SOP file is sufficient to support a new error code — no code changes needed in the orchestrator.

📄 SOP_PDE_781.txt

SOP – PDE Error Code 781
-----------------------
Error: Provider ID missing or invalid

Resolution Steps:
1. Identify claims with error code 781
2. Validate Provider ID vs CMS master provider file
3. If missing → retrieve from RxClaim system
4. If invalid → correct via CMS provider mapping
5. Update claim record with corrected Provider ID
6. Mark claim as READY_FOR_REPROCESS
7. Include in PDE resubmission flat file
8. Document correction in audit log

Output:
- Updated claim record
- Corrected PDE flat file
- Claim ready for CMS resubmission

📄 SOP_PDE_935.txt

SOP – PDE Error Code 935
-----------------------
Error: Claim already adjudicated

Resolution Steps:
1. Identify claims with error code 935
2. Compare Received Date vs Adjudication Timestamp
3. If adj_ts newer → mark ALREADY_PROCESSED
4. Do NOT reprocess the claim
5. Include claim ONLY in reporting output
6. Exclude from PDE resubmission file
7. Document decision for audit compliance

Output:
- Claim excluded from reprocessing
- Compliance report generated
- Audit record updated
💡 Extensibility: To add support for a new CMS error code (e.g. 456), simply create sop/SOP_PDE_456.txt with the resolution steps. The dynamic orchestrator will automatically read it and derive the correct pipeline without any code changes.

ReAct Pattern Explained

ReAct (Reasoning + Acting) is the LLM agent pattern used throughout this system. Each agent runs in a loop: it reasons about what to do next, picks a tool, observes the result, then reasons again — until it reaches a final answer.

 ┌─────────────────────────────────────────────────────┐
 │              ReAct Loop (per agent)                 │
 │                                                     │
 │  ┌────────────────────────────────────────────────┐ │
 │  │  1. THOUGHT  (GPT-4o reasons about next step)  │ │
 │  │     "I need to load the SOP first..."          │ │
 │  └─────────────────────┬──────────────────────────┘ │
 │                        │                            │
 │  ┌─────────────────────▼──────────────────────────┐ │
 │  │  2. ACTION   (calls a tool)                    │ │
 │  │     tool_load_sop("781")                       │ │
 │  └─────────────────────┬──────────────────────────┘ │
 │                        │                            │
 │  ┌─────────────────────▼──────────────────────────┐ │
 │  │  3. OBSERVATION  (tool returns result)         │ │
 │  │     "SOP for 781: validate provider ID..."     │ │
 │  └─────────────────────┬──────────────────────────┘ │
 │                        │                            │
 │                        └──────────── LOOP ──────────┤
 │                        (repeat until final answer)  │
 │                                                     │
 │  ┌────────────────────────────────────────────────┐ │
 │  │  4. FINAL ANSWER  (structured JSON response)   │ │
 │  │     { "next_agent": "DONE", "decision": ... }  │ │
 │  └────────────────────────────────────────────────┘ │
 └─────────────────────────────────────────────────────┘

Why ReAct for this use case?

PropertyBenefit
Dynamic tool selectionThe LLM decides which tools to call based on context — no hardcoded routing
Observable reasoningEvery thought and tool call is captured in the trace, giving full auditability
Error recoveryIf a tool returns an unexpected result, the LLM can adapt its next step
SOP complianceThe LLM reads official SOPs and follows them — easily updated by editing text files
ScalabilityNew error codes or agents can be added without touching orchestration logic
